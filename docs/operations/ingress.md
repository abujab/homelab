# Ingress Operations

---

## Purpose

This document describes operational procedures for the HomeLab ingress
foundation.

## Scope

This document covers:

- deploying Traefik
- deploying the ingress test application
- publishing the Pi-hole Web UI
- onboarding future internal Web applications
- validating DNS and HTTPS routing
- basic troubleshooting
- rollback

Certificate lifecycle operations are covered by the certificate operations
document. This document does not cover public internet exposure or production
application onboarding.

## Background

Traefik is the standard Kubernetes Ingress Controller for HomeLab.

It receives one MetalLB LoadBalancer IP and routes HTTPS traffic to Kubernetes
Services based on hostnames. HTTP requests are redirected to HTTPS.

## Architecture / Implementation

### Deploy Traefik

Add and update the official Traefik Helm repository:

```bash
helm repo add traefik https://traefik.github.io/charts
helm repo update traefik
```

Create the namespace:

```bash
kubectl --kubeconfig ansible/kubeconfig apply -f kubernetes/platform/ingress/namespace.yaml
```

Install or upgrade Traefik:

```bash
helm upgrade --install traefik traefik/traefik \
  --version 41.0.2 \
  --namespace ingress \
  --values kubernetes/platform/ingress/values.yaml \
  --kubeconfig ansible/kubeconfig
```

Wait for rollout:

```bash
kubectl --kubeconfig ansible/kubeconfig rollout status deployment/traefik -n ingress --timeout=300s
```

### Deploy the test application

Apply the test application manifests:

```bash
kubectl --kubeconfig ansible/kubeconfig apply -f kubernetes/platform/ingress/test-app/deployment.yaml
kubectl --kubeconfig ansible/kubeconfig apply -f kubernetes/platform/ingress/test-app/service.yaml
kubectl --kubeconfig ansible/kubeconfig apply -f kubernetes/platform/ingress/test-app/ingress.yaml
```

Wait for rollout:

```bash
kubectl --kubeconfig ansible/kubeconfig rollout status deployment/whoami -n ingress --timeout=300s
```

### Validate ingress

Check Traefik pods:

```bash
kubectl --kubeconfig ansible/kubeconfig get pods -n ingress -l app.kubernetes.io/name=traefik -o wide
```

Expected result:

- two Traefik pods are `Running`
- the pods are preferably scheduled on different nodes

Check the Traefik Service:

```bash
kubectl --kubeconfig ansible/kubeconfig get svc -n ingress traefik
```

Expected external IP:

```text
192.168.68.201
```

Check Ingress resources:

```bash
kubectl --kubeconfig ansible/kubeconfig get ingress -A
```

Expected host:

```text
test.home.arpa
```

Check DNS:

```bash
dig @192.168.68.200 test.home.arpa +short
```

Expected answer:

```text
192.168.68.201
```

If the management workstation or browser client is configured to use Pi-hole as
its DNS resolver, the plain resolver check should also work:

```bash
dig test.home.arpa +short
```

Check HTTPS routing:

```bash
curl https://test.home.arpa
```

Expected result:

```text
Hostname:
```

The response should be generated by the `whoami` test application.

### Deploy Pi-hole Web ingress

Apply the Pi-hole resources through the existing kustomization:

```bash
kubectl --kubeconfig ansible/kubeconfig apply \
  -k kubernetes/platform/networking/pihole
```

The resulting exposure model is:

```text
pihole Service      LoadBalancer  192.168.68.200  53/TCP,53/UDP
pihole-web Service  ClusterIP                     80/TCP
pihole Ingress      Traefik       pihole.home.arpa
```

Validate the complete model:

```bash
scripts/validate-pihole-exposure.sh
```

Pi-hole v6 exposes its dashboard at:

```text
https://pihole.home.arpa/admin/
```

The root path currently returns `403 Forbidden`, so no path rewrite or
application redirect is configured. This avoids breaking Pi-hole assets, API
requests and login redirects.

### Publish a new internal Web application

1. Create a ClusterIP Service for the application's HTTP port.
2. Add a Pi-hole local DNS record mapping the application hostname to
   `192.168.68.201`.
3. Create a cert-manager `Certificate` in the application namespace using
   `ClusterIssuer/homelab-server-ca` and a DNS SAN for the hostname.
4. Create a standard Kubernetes Ingress using `ingressClassName: traefik`, the
   Certificate's TLS Secret and the ClusterIP Service backend.
5. Apply the resources and wait for `Certificate` to report `Ready=True`.
6. Validate DNS, HTTP redirection, trusted HTTPS and the backend route.

Do not allocate a dedicated LoadBalancer IP for an ordinary Web UI. Document an
exception when the application requires a non-HTTP protocol, end-to-end TLS,
mTLS or TLS passthrough.

If the client is not yet using Pi-hole for DNS, validate Traefik routing
directly with the expected hostname and Root CA:

```bash
curl --resolve test.home.arpa:443:192.168.68.201 \
  --cacert "${HOMELAB_PKI_DIR}/root/certs/homelab-root-ca.crt" \
  https://test.home.arpa/
```

Check HTTP redirect:

```bash
curl --resolve test.home.arpa:80:192.168.68.201 \
  -I http://test.home.arpa/
```

Check certificate issuance for an application namespace:

```bash
kubectl --kubeconfig ansible/kubeconfig get certificate,certificaterequest \
  -n networking
kubectl --kubeconfig ansible/kubeconfig describe certificate \
  pihole-home-arpa -n networking
```

### Roll back Pi-hole Web ingress

DNS availability at `192.168.68.200` and the `pihole-config` PVC do not depend
on the Ingress. If the ingress path fails:

1. Add the original `http` port entry back to the existing `pihole` Service.
2. Apply `service.yaml` in place; do not delete the Service or PVC.
3. Restore the deployment's local record to
   `pihole.home.arpa -> 192.168.68.200` and wait for rollout.
4. Delete only `Ingress/pihole` if it conflicts with recovery; the Certificate
   may remain.
5. Confirm UDP and TCP DNS remain available through `192.168.68.200`.

Rollback does not require PKI regeneration. After the ingress fault is fixed,
reapply the repository kustomization to restore the desired architecture.

## Design Decisions

Traefik is operated through Helm because the official chart owns the related
Deployment, Service, RBAC and IngressClass resources.

The dashboard is not exposed. Use `kubectl port-forward` only for temporary
local troubleshooting.

## Best Practices

- keep Traefik chart upgrades deliberate and reviewed
- validate DNS before debugging HTTP routing
- validate the IngressClass when routes are not picked up
- keep backend Services as `ClusterIP` unless they truly require direct LAN
  exposure
- avoid exposing the dashboard until authentication is implemented
- validate certificate readiness before debugging Traefik routing
